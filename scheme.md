# 聖經研讀資料整理工程設計說明（scheme.md）

## 一、總體目標
建立一套可擴展、結構清晰、適合 Obsidian 使用的聖經研讀資料庫，以《但以理書》為試點，之後可依相同模式整理其他書卷。資料庫將經文、註解、解說、背景資料、綱要、拾穗與交叉參照等分類存放，並以中文章節為檔案命名單位，方便在 Obsidian 中進行雙向連結與主題探索。

## 二、資料夾結構
以下為預設根目錄 `C:\Obsidian\Hermes\scripture\` 下的建議結構（亦可根據實際需求微調）：

```
scripture/
├── scheme.md                     ← 本檔案，說明整體設計與使用說明
├── 【書名】/                     ← 每卷聖經獨立資料夾（以中文書名命名）
│   ├── 經文/                     ← 放置該卷每章的淨化經文
│   │   └── 第x章.md              ← x 為章數，例如 第1章.md、第2章.md
│   ├── 註解/                     ← 各章節的傳統或學術註解（如查經資料大全）
│   │   └── 第x章.md
│   ├── 解說/                     ← 綜合多方資料的個人研讀筆記（含神學反思）
│   │   └── 第x章.md
│   ├── 拾穗/                     ← 各章節的背景資料、歷史補充、文化說明等
│   │   └── 第x章.md
│   ├── 背景/                     ← 整卷的背景資料（如作者、寫作時期、原意等）
│   │   └── 全書背景.md
│   ├── 綱要/                     ← 整卷的大綱與主題概覽
│   │   └── 全書綱要.md
│   └── 交叉參照/                 ← 各章節的內部或跨卷交叉參照整理
│       └── 第x章.md
├── 人物/                         ← 跨書卷的聖經人物條目（可跨卷重複使用）
│   └── 【人名】.md               │ 例如 但以理.md、尼布甲尼撒.md、沙得拉.md
├── 地點/                         ← 跨書卷的聖經地點條目
│   └── 【地名】.md               │ 例如 巴比倫.md、耶路撒冷.md、杜拉平原.md
└── 主題/                         ← 跨書卷的神學或歷史主題條目
    └── 【主題】.md               │ 例如 神的同在.md、烈火窑.md、信仰的堅持.md
```

> **說明**  
> - 每卷書皆以資料夾方式隔離，避免檔名衝突。  
> - 人物、地點、主題三個資料夾為「交叉引用庫」，供所有書卷共用，以減少重複建立相同條目。  
> - 若某卷書無需某類資料（如背景、綱要），可保留該資料夾為空或直接刪除，不影響整體結構。

## 三、檔案命名與連結定位規則
1. **章節檔**：各資料夾內仍使用「中文章節號 + .md」格式，例如 `第1章.md`、`第10章.md`。  
   - 但因不同書卷、不同資料類型都會有 `第1章.md`，**任何跨檔 wiki-link 必須使用完整路徑與 alias**。
   - 正確：`[[何西阿書/經文/第1章|何西阿書 第1章]]`
   - 正確：`[[何西阿書/註解/第1章|何西阿書 第1章 註解]]`
   - 錯誤：`[[第1章]]`、`[[何西阿書 第1章]]`（除非真的有此檔案）
2. **人物／地點／主題條目**：使用常見中文名稱或標準譯名，例如 `沙得拉.md`、`巴比倫.md`、`神的同在.md`。
3. **同名消歧**：若人名、地名、王名可能重複，檔名必須消歧，顯示時用 alias。
   - 例：`[[約阿施（以色列王）|約阿施]]`
   - 例：`[[約阿施（猶大王）|約阿施]]`
4. **書卷與人物分開**：書卷一律用完整書名，人物用人名。
   - 書卷：`[[何西阿書]]`
   - 人物：`[[何西阿]]`
   - 避免：`[[何西阿]]書`
5. **別名與縮寫**：人物有多名稱號時，以最精準檔名為主，顯示文字用 alias。
   - 例：`[[耶羅波安二世|耶羅波安]]`
   - 例：`[[烏西亞|亞撒利雅]]`
6. **特別檔案**：全書背景、全書綱要分別命名為 `全書背景.md`、`全書綱要.md`，置於各書卷對應資料夾底下。

## 四、每篇 markdown 文章的格式模板
以下為各類檔案的建議結構（可依需求增減），所有標題使用 `##` 或 `###` 以保持層級清晰，適合 Obsidian 的大綱面板與反向連結。

### 1. 經文檔（例如 `但以理書/經文/第3章.md`）

```
2. 經文內容……  
3. 經文內容……

```

## 五、來源資料的標註方式
1. 在每篇 markdown 檔案開頭的「## 資料來源」小節中，列出所使用的主要來源，例如：
   - 查經資料大全：
   - KingComments：
   - Bible Hub Study Bible：
   - 其它（如個人筆記、講道稿、學術論文）。
2. 在具體內容中，若直接引用某來源的觀點，可使用腳註形式或括註說明，例如：
   > （查經資料大全，但以理書第三章註解）
   > （KingComments，Daniel 3）
3. 若是將多個來源的資訊做整合與摘要，則在該段落結尾處加入綜合來源說明，例如：
   > （綜合自查經資料大全與 Bible Hub Study Bible）

## 六、擴展至其他書卷的做法
1. **新增書卷**：在 `scripture/` 底下建立新資料夾，使用該書的中文全名（例如 `約翰福音`、`啟示錄`）。  
2. **複製結構**：參照本 scheme.md 中的資料夾與檔案命名規則，建立對應的子資料夾（經文、註解、解說、拾穗、背景、綱要、交叉參照）。  
3. **共用資料庫**：人物、地點、主題三個資料夾為全域共用，新書卷產生的條目直接放入這三個資料夾中，若已存在則只需在對應章節的交叉參照中加上 `[[已存在條目]]` 的連結。  
4. **格式統一**：所有新書卷的 markdown 檔案皆採用上述格式模板，以確保觀感一致且連結正常。  
5. **定期檢查**：使用 Obsidian 的「破損連結」與「未連結檔案」功能，及時修正遺漏或錯誤的連結。

## 七、來源網站 URL 模式與抓取規則

### 1. 經文：cnbible.com 和合本

| 類型 | URL |
|---|---|
| 繁體 CUV | `https://cnbible.com/cu/{book_slug}/{章}.htm` |
| 簡體 CUS | `https://cnbible.com/cus/{book_slug}/{章}.htm` |

例：`hosea/1.htm`、`1_samuel/1.htm`

規則：
- `{book_slug}` 使用網站 slug，不是中文書名，也不一定是一般英文全名。
- 多字書卷常用底線，例如 `1_samuel`。
- 抓取後常有中文字間空格，例如「當 烏 西 雅」，必須後處理移除。
- 若抓到目錄頁、空頁、404 或非經文本體，視為失敗。

---

### 2. 查經資料大全：ccbiblestudy.org

#### 2.1 目錄頁

| 類型 | URL |
|---|---|
| 舊約目錄 | `https://www.ccbiblestudy.org/Old%20Testament/{卷代碼}/{卷號}index-T.htm` |
| 新約目錄 | `https://www.ccbiblestudy.org/New%20Testament/{卷代碼}/{卷號}index-T.htm` |

例：`28Hosea/28index-T.htm`、`40Matt/40index-T.htm`

#### 2.2 逐章資料

| 類型 | URL |
|---|---|
| 舊約註解 | `https://www.ccbiblestudy.org/Old%20Testament/{卷代碼}/{卷號}CT{章2位}.htm` |
| 新約註解 | `https://www.ccbiblestudy.org/New%20Testament/{卷代碼}/{卷號}CT{章2位}.htm` |
| 舊約拾穗 | `https://www.ccbiblestudy.org/Old%20Testament/{卷代碼}/{卷號}GT{章2位}.htm` |
| 新約拾穗 | `https://www.ccbiblestudy.org/New%20Testament/{卷代碼}/{卷號}GT{章2位}.htm` |

例：
- `28Hosea/28CT01.htm`
- `28Hosea/28GT01.htm`
- `27Daniel/27CT01.htm`
- `01Gen/01CT01.htm`
- `40Matt/40CT01.htm`

規則：
- `{卷代碼}` 不是固定英文全名，必須以該書目錄頁為準。
- `{卷號}` 是聖經卷序兩位數，例如創世記 `01`、但以理書 `27`、何西阿書 `28`、馬太福音 `40`。
- `{章2位}` 必須補零，例如第 1 章為 `01`。
---
### 3. KingComments 解說

`https://www.kingcomments.com/en/bible-studies/{KC_slug}/{章}`

例：`Hos/1`、`Dan/1`、`Gen/1`、`1Sam/1`、`Song/1`

規則：
- `{KC_slug}` 以 KingComments 網站實際連結為準，不自行猜。
- 若逐章頁不存在，先回到該卷目錄或搜尋該書縮寫。

---

### 4. BibleHub Study

`https://biblehub.com/study/{book_slug}/{章}.htm`

例：`hosea/1.htm`、`1_samuel/1.htm`

規則：
- `{book_slug}` 使用 BibleHub slug，不一定等於普通英文書名。
- BibleHub 可作為 study notes、cross references、commentary 補充來源。

---

### 5. 抓取優先順序

1. 先取經文：cnbible CUV。
2. 再取查經資料大全：註解 CT、拾穗 GT。
3. 再取 KingComments 解說。
4. 再取 BibleHub Study 作補充。
5. 若已收集資料足夠，不為了補條目而額外搜尋新來源。

### 6. 抓取成功判準

抓取後必須檢查內容是否為本體資料：

- 經文頁必須含有該章經文。
- 註解頁必須含有「逐節詳解」或類似註解內容。
- 拾穗頁必須含有背景、字義、歷史或補充資料。
- 目錄頁、廣告頁、空內容、錯章、亂碼、404、Internal Error 都視為失敗。

---

## 八、資料清理與來源使用規則

1. **只用已收集資料**：除非使用者明確要求，整理、補條目、補 wiki-link 時不得自行上網新增來源。
2. **保留來源，不保留雜訊**：刪除網站導航、廣告、版權區塊、重複目錄、無關連結。
3. **保持來源邊界**：註解、拾穗、解說可整合，但不可把不同來源的觀點混成單一來源聲稱。
4. **標明整合**：若一段內容綜合多來源，段末標註「綜合自……」。
5. **不硬補**：資料不足時，放入「待確認事項」或建立候選條目，不用外部薄資料湊完整條目。
6. **繁體中文輸出**：所有輸出、檔名、標題與條目內容一律使用繁體中文。

## 九、wiki-link 規則（知識節點優先）

### 1. 核心原則

wiki-link 的目的不是把所有名詞都標起來，而是建立一個可逐章累積、跨卷串聯的知識網絡。

判斷標準：

> 這個詞是否值得成為日後查經、回顧、比較、跨章串聯、跨書卷串聯時會用到的知識節點。

### 2. 必須建立 wiki-link 的類型

- 人物：`[[何西阿]]`、`[[歌篾]]`、`[[耶戶]]`
- 地點：`[[耶斯列]]`、`[[耶斯列平原]]`、`[[亞述]]`
- 國家／群體：`[[以色列]]`、`[[猶大]]`
- 書卷：`[[何西阿書]]`、`[[羅馬書]]`、`[[彼得前書]]`
- 歷史事件：`[[北國以色列滅亡]]`、`[[亞述入侵]]`
- 王朝或政治背景：`[[耶戶王朝]]`、`[[耶羅波安二世以後的北國君王]]`
- 神學主題：`[[屬靈淫亂]]`、`[[偶像崇拜]]`、`[[審判後的復興]]`
- 聖經母題：`[[婚姻作為盟約隱喻]]`、`[[先知的象徵行動]]`
- 原文字義或名字神學：`[[耶斯列的雙關]]`、`[[羅路哈瑪]]`、`[[羅阿米]]`
- 解經爭議：`[[歌篾何時成為淫婦]]`
- 新舊約互文：`[[非我民到神的兒子]]`、`[[外邦人得救]]`

### 3. 普通詞處理原則

普通名詞本身通常不建立 wiki-link。若該詞在本章中承載象徵、隱喻、審判、復興、盟約或救恩意義，應連結「完整概念短語」，不是單一普通詞。

- 不連 `[[弓]]`，而連 `[[折斷以色列的弓]]`
- 不連 `[[馬匹]]`，而連 `[[不靠軍事力量得救]]`
- 不連 `[[海沙]]`，而連 `[[亞伯拉罕之約的海沙應許]]`
- 不連 `[[兒女]]`，而連 `[[象徵性命名]]`
- 不連 `[[淫亂]]`，而連 `[[屬靈淫亂]]`

### 4. 關於「神、耶和華、主」

不要機械式地把每一個「神」、「耶和華」、「主」都連成同一個 wiki-link。若該詞與特定神學概念、盟約行動、啟示、屬性或救恩作為有關，建立主題型連結。

例如：

- `[[耶和華的話臨到]]`
- `[[耶和華是以色列的丈夫]]`
- `[[神的公義與慈愛]]`
- `[[神的憐憫]]`
- `[[神的棄絕非永遠]]`
- `[[不作你們的神]]`
- `[[我是自有永有的]]`

### 5. 連結密度

1. 同一段落中，同一詞第一次出現時建立 link 即可。
2. 標題、表格欄位、主題清單、整合觀察、待確認事項中的關鍵詞優先建立 link。
3. 經文檔偏保守：主要連人物、地點、國家、書卷、核心主題。
4. 註解、解說、拾穗檔較積極：可連神學主題、歷史事件、象徵短語、解經爭議。
5. 不要為了連結而破壞可讀性。

### 6. 禁止與避免

- 禁止把不存在的檔名當成 link，例如 `[[何西阿書 第1章]]`。
- 禁止同一段落重複密集連同一詞。
- 禁止為了讓 verify 通過而大量建立薄弱條目。
- 避免把普通字詞、單純動詞、單純形容詞建成條目。

## 十、條目建立、候選條目與豐富化規則

### 1. 條目狀態

所有人物、地點、主題條目可分成兩種狀態：

| 狀態 | 用途 |
|---|---|
| `formal` | 已有足夠資料，可作正式條目 |
| `candidate` | 目前資料不足，但值得保留為日後擴充節點 |

建議使用 frontmatter：

```md
---
type: 主題
status: candidate
source_scope: collected_only
aliases: []
---
```

### 2. 正式條目

人物、地點、國家、核心主題、重要神學概念，應建立正式條目。正式條目至少包含：

- 基本資料
- 聖經出現
- 與本書卷的關聯
- 神學意義
- 相關條目
- 資料來源
- 待確認事項

### 3. 候選條目

若該詞是解經爭議、象徵短語、互文主題，但目前資料不足，先建立候選條目，不要外找資料硬補。

```md
---
type: 解經爭議
status: candidate
source_scope: collected_only
aliases: []
---

# 歌篾何時成為淫婦

## 類型
解經爭議

## 出現位置
- [[何西阿書/註解/第1章|何西阿書 第1章 註解]]
- [[何西阿書/解說/第1章|何西阿書 第1章 解說]]

## 目前資料
- 有觀點認為[[歌篾]]婚前已是淫婦。
- 有觀點認為[[歌篾]]婚後才成為淫婦。
- 有觀點認為此敘事可能是寓言或象徵敘事。

## 相關條目
- [[歌篾]]
- [[屬靈淫亂]]
- [[婚姻作為盟約隱喻]]

## 待補充
- 目前只根據已收集資料整理，日後有更多來源時再擴充。
```

### 4. 防止條目無限擴散

補建條目時，條目內仍可使用 wiki-link，但必須遵守：

1. 優先連既有條目。
2. 新增 link 前先判斷是否真的值得成為知識節點。
3. 不為了新條目中的每個普通詞再建立下一層條目。
4. 若資料不足，使用候選條目或純文字，不硬生新條目。
5. 本輪章節整理以「該章直接相關節點」為主，不無限擴張到二階、三階背景。

### 5. 已有條目豐富化

若條目已存在，不要跳過。應根據本章內容補充：

- 「聖經出現」新增本章引用
- 「與本書卷的關聯」新增本章角色
- 「神學意義」新增本章觀察
- 「相關條目」新增本章形成的新連結

## 十一、強制連結驗證流程

### 1. 每章完成後必跑

每完成一章，必須執行：

1. 掃描該章四類檔案：經文、註解、拾穗、解說。
2. 掃描本輪新增或修改的人物、地點、主題條目。
3. 比對 `人物/`、`地點/`、`主題/` 及各書卷資料夾中所有 markdown 檔。
4. 輸出缺失清單 `missing_entities.txt`。
5. 補建正式條目或候選條目。
6. 再跑驗證，直到 0 破損連結。
7. 通過後才可 commit + push。

### 2. verify_links.py 必須支援

- 掃描指定書卷或全部書卷。
- 遞迴掃描 `人物/`、`地點/`、`主題/` 與各書卷資料夾。
- 解析 `[[頁名]]` 與 `[[頁名|顯示文字]]`。
- 解析含路徑連結，例如 `[[何西阿書/經文/第1章|何西阿書 第1章]]`。
- 忽略外部 URL、程式碼區塊、frontmatter 中不該檢查的欄位。
- 輸出 JSON 報告與人類可讀報告。
- 依上下文推斷建議類型：人物、地點、主題、書卷、候選。

### 3. 缺失處理

缺失 link 的處理順序：

1. 如果是拼字或 alias 錯誤，修正原 link。
2. 如果已有同義條目，改用既有條目與 alias。
3. 如果是重要實體或核心主題，建立正式條目。
4. 如果資料不足但值得保留，建立候選條目。
5. 如果不值得成為節點，移除 wiki-link，改為純文字。

### 4. 禁止事項

- 禁止跳過驗證直接 commit。
- 禁止依賴 agent 記憶判斷「應該沒問題」。
- 禁止用薄弱外部搜尋結果湊條目。
- 禁止為了消除 missing links 而無限新增低價值條目。
- 禁止使用未經使用者允許的新來源補充內容。

## 十二、章節處理標準流程

每整理一章，依序執行：

1. 建立或確認資料夾結構。
2. 抓取並清理經文。
3. 抓取並清理註解。
4. 抓取並清理拾穗。
5. 抓取並整理解說來源。
6. 整合觀察：只根據已收集資料，不加入未來源化內容。
7. 建立或強化 wiki-link。
8. 補建正式條目或候選條目。
9. 更新既有條目。
10. 執行 verify_links.py。
11. 修正缺失直到 0 破損。
12. 在章節「待確認事項」列出仍不確定的問題。
13. 輸出整理報告。
14. commit + push。

整理報告至少包含：

- 本章完成的檔案
- 使用來源
- 新增 wiki-link
- 新增正式條目
- 新增候選條目
- 更新既有條目
- verify_links.py 結果
- 待確認事項

## 十三、維護與版本規則

- 本檔案是最高規則。既有書卷慣例只能參考；若與本檔案衝突，以本檔案為準。
- 每次修改規則，必須在本檔案更新原因與範例。
- 建議每完成一卷書後，檢查：破損連結、孤立檔案、重複條目、過度細碎主題。
- 若某規則導致條目爆炸或 Agent 行為不穩，優先收緊 wiki-link 判斷標準，而不是放寬驗證流程。
- 中文輸出一律使用繁體中文。
