# 聖經研讀資料整理工程設計說明（scheme.md）

## 一、總體目標
建立一套可擴展、結構清晰、適合 Obsidian 使用的聖經研讀資料庫，以《但以理書》為試點，之後可依相同模式整理其他書卷。資料庫將經文、註解、解說、背景資料、綱要、拾穗與交叉參照等分類存放，並以中文章節為檔案命名單位，方便在 Obsidian 中進行雙向連結與主題探索。

## 二、資料夾結構
以下為預設根目錄 `C:\Obsidian\Hermes\scripture\` 下的建議結構（亦可根據實際需求微調）：

```
scripture/
├── scheme.md                     ← 本檔案，說明整體設計與使用說明
├── 【書名】/                     ← 每卷聖經獨立資料夾（以中文書名命名）
│   ├── 經文/                     ← 放置該卷每章的淨化經文
│   │   └── 第x章.md              ← x 為章數，例如 第1章.md、第2章.md
│   ├── 註解/                     ← 各章節的傳統或學術註解（如查經資料大全）
│   │   └── 第x章.md
│   ├── 解說/                     ← 綜合多方資料的個人研讀筆記（含神學反思）
│   │   └── 第x章.md
│   ├── 拾穗/                     ← 各章節的背景資料、歷史補充、文化說明等
│   │   └── 第x章.md
│   ├── 背景/                     ← 整卷的背景資料（如作者、寫作時期、原意等）
│   │   └── 全書背景.md
│   ├── 綱要/                     ← 整卷的大綱與主題概覽
│   │   └── 全書綱要.md
│   └── 交叉參照/                 ← 各章節的內部或跨卷交叉參照整理
│       └── 第x章.md
├── 人物/                         ← 跨書卷的聖經人物條目（可跨卷重複使用）
│   └── 【人名】.md               │ 例如 但以理.md、尼布甲尼撒.md、沙得拉.md
├── 地點/                         ← 跨書卷的聖經地點條目
│   └── 【地名】.md               │ 例如 巴比倫.md、耶路撒冷.md、杜拉平原.md
└── 主題/                         ← 跨書卷的神學或歷史主題條目
    └── 【主題】.md               │ 例如 神的同在.md、烈火窑.md、信仰的堅持.md
```

> **說明**  
> - 每卷書皆以資料夾方式隔離，避免檔名衝突。  
> - 人物、地點、主題三個資料夾為「交叉引用庫」，供所有書卷共用，以減少重複建立相同條目。  
> - 若某卷書無需某類資料（如背景、綱要），可保留該資料夾為空或直接刪除，不影響整體結構。

## 三、檔案命名規則
1. **章節檔**：一律使用「中文章節號 + .md」格式，例如 `第1章.md`、`第10章.md`。  
   - 若書卷有「上、下篇」或「第一部、第二部」等大分部，可在資料夾中再加一層，例如 `但以理書/經文/上篇/第1章.md`，但試作品階段保持單層簡單。
2. **人物/地點/主題**：使用實體的常見中文名稱或標準譯名，例如 `沙得拉.md`、`巴比倫.md`、`神的同在.md`。  
   - 若有重名稱號稱或外號較常見，可在檔案內加上別名（見下文格式）。
3. **特別檔案**：全書背景、全書綱要分別命名為 `全書背景.md`、`全書綱要.md`，置於各書卷對應資料夾底下。

## 五、來源資料的標註方式
1. 在每篇 markdown 檔案開頭的「## 資料來源」小節中，列出所使用的主要來源，例如：
   - 查經資料大全：
   - KingComments：
   - Bible Hub Study Bible：
   - 其它（如個人筆記、講道稿、學術論文）。
2. 在具體內容中，若直接引用某來源的觀點，可使用腳註形式或括註說明，例如：
   > （查經資料大全，但以理書第三章註解）
   > （KingComments，Daniel 3）
3. 若是將多個來源的資訊做整合與摘要，則在該段落結尾處加入綜合來源說明，例如：
   > （綜合自查經資料大全與 Bible Hub Study Bible）

## 六、擴展至其他書卷的做法
1. **新增書卷**：在 `scripture/` 底下建立新資料夾，使用該書的中文全名（例如 `約翰福音`、`啟示錄`）。  
2. **複製結構**：參照本 scheme.md 中的資料夾與檔案命名規則，建立對應的子資料夾（經文、註解、解說、拾穗、背景、綱要、交叉參照）。  
3. **共用資料庫**：人物、地點、主題三個資料夾為全域共用，新書卷產生的條目直接放入這三個資料夾中，若已存在則只需在對應章節的交叉參照中加上 `[[已存在條目]]` 的連結。  
4. **格式統一**：所有新書卷的 markdown 檔案皆採用上述格式模板，以確保觀感一致且連結正常。  
5. **定期檢查**：使用 Obsidian 的「破損連結」與「未連結檔案」功能，及時修正遺漏或錯誤的連結。

## 七、來源網站 URL 模式（2026-06-24 實測紀錄）

以下 URL 模式經本次《何西阿書》實際抓取驗證，**失敗的組合已標註**，避免下次重複試錯：

### 1. 經文（cnbible.com 和合本）

| 語言 | ✅ 正確 URL | ❌ 失敗 URL |
|------|-------------|-------------|
| 繁體中文 | `https://cnbible.com/cu/{英文書名}/{章}.htm` | `https://cnbible.com/{中文書名}/{章}.htm`（回目錄頁） |
| 簡體中文 | `https://cnbible.com/cus/{英文書名}/{章}.htm` | `https://cnbible.com/{中文書名}.htm`（404） |

**英文書名對照**：Hosea（何西阿書）、Genesis（創世記）、Daniel（但以理書）等。

**注意**：逕行 web_extract 回傳內容有**字間空格**（如「當 烏 西 雅」），需後處理移除。

### 2. 註解（ccbiblestudy.org 查經資料大全）

| 語言 | ✅ 正確 URL | ❌ 失敗 URL |
|------|-------------|-------------|
| 繁體中文 | `https://www.ccbiblestudy.org/Old%20Testament/{編號}{書名}/{編號}CT{章}.htm` | `https://ccbiblestudy.org/commentaries/{編號}-{書名}/01.htm`（404） |

**編號對照**：28Hosea（何西阿書）、28Daniel（但以理書）。

**指令碼**：`{編號}CT{章}.htm` = 註解（CT = Commentary Traditional）；`{編號}BT{章}.pdf` = 黃迦勒 PDF；`{編號}GT{章}.htm` = 拾穗。

### 3. 解說（kingcomments.com）

**正確 URL**：`https://www.kingcomments.com/en/bible-studies/{英文書名縮寫}/{章節阿拉伯數字}`

**英文書名縮寫對照**：Hos（何西阿書）、Dan（但以理書）、Gen（創世記）等。

**用途**：逐章解說，適合做「解說」章節的主要來源。

### 4. 研讀資料（biblehub.com）

**正確 URL 基底**：`https://biblehub.com/study/`

**逐章 URL**：`https://biblehub.com/study/{英文書名}/{章}.htm`

**用途**：提供 study notes、cross references、commentary 等資料，適合做「解說」的補充來源。

### 5. 拾穗（ccbiblestudy.org）

| 語言 | ✅ 正確 URL |
|------|-------------|
| 繁體中文 | `https://www.ccbiblestudy.org/Old%20Testament/{編號}{書名}/{編號}GT{章}.htm` |

**指令碼**：`{編號}GT{章}.htm` = 拾穗（GT = Gleanings Traditional）。

### 6. 抓取流程建議

1. 先用 `web_search` 搜尋 `{書名} 第{章} 經文 和合本 cnbible` 確認正確 URL
2. 用 `web_extract` 抓取 HTML 內容
3. 若 web_extract 失敗或內容為空，嘗試 `web_search` 找替代頁面
4. 若需中文但只有英文來源，直接抓取英文內容再於整理時轉譯
5. 每章抓取後**立即檢查**內容是否包含經文/註解本體（而非僅導航目錄）

2. **不依賴事後補建**：不要等 verify_links 掃描後再加，要在寫的時候就加好
3. **新實體立刻建檔**：遇到新的實體名，無論是否在人物/地點/主題資料夾中已存在，都要在寫完當下建立或確認對應檔案
4. **已有實體豐富化**：若該實體已存在，**不要跳過**，應根據本章內容補充該實體的「聖經出現」紀錄、神學意義、或與本章的關聯性。特別是核心實體（如[[何西阿]]、[[以色列]]、[[耶和華]]、[[巴力]]等），每章都會提到，應逐章累積其內容深度
5. **verify_links.py 是安全網**：掃描只能發現「漏網之魚」，不能取代撰寫時的自覺

### 格式
- 標準格式：`[[實體名]]`
- 別名顯示：`[[全名|顯示名]]`
- 禁止：裸體文字不加 link（如寫「何西阿」而非 `[[何西阿]]`）

### 範例
正確：`[[何西阿]]去娶了[[滴拉音]]的女兒[[歌篾]]`
錯誤：`何西阿去娶了滴拉音的女兒歌篾`

### wiki-link 判斷標準補充：由「實體連結」擴展為「知識節點連結」

本知識庫的 wiki-link 不只用來標記人物、地點、國家等實體，也用來建立可跨章、跨卷、跨主題探索的知識節點。

判斷某個詞是否需要建立 wiki-link，不只看它是否為「實體」，更要看它是否具備以下任一功能：

1. 日後查經時會反覆回來查詢
    
2. 能串聯不同章節或不同書卷
    
3. 能代表本章的神學主題
    
4. 能代表本章的歷史背景
    
5. 能代表某個重要象徵、隱喻或母題
    
6. 能代表原文字義、名字含義、雙關語
    
7. 能代表解經爭議或不同解釋立場
    
8. 能成為人物、地點、主題條目的補充入口
    

#### 必須建立 wiki-link 的類型

- 人物：例如 [[何西阿]]、[[歌篾]]、[[耶戶]]
    
- 地點：例如 [[耶斯列]]、[[耶斯列平原]]、[[亞述]]
    
- 國家／群體：例如 [[以色列]]、[[猶大]]
    
- 書卷：例如 [[何西阿書]]、[[羅馬書]]、[[彼得前書]]
    
- 歷史事件：例如 [[北國以色列滅亡]]、[[亞述入侵]]
    
- 王朝或政治背景：例如 [[耶戶王朝]]、[[耶羅波安二世以後的北國君王]]
    
- 神學主題：例如 [[屬靈淫亂]]、[[偶像崇拜]]、[[審判後的復興]]
    
- 聖經母題：例如 [[婚姻作為盟約隱喻]]、[[先知的象徵行動]]
    
- 原文字義或名字神學：例如 [[耶斯列的雙關]]、[[羅路哈瑪]]、[[羅阿米]]
    
- 解經爭議：例如 [[歌篾何時成為淫婦]]
    
- 新舊約互文：例如 [[非我民到神的兒子]]、[[外邦人得救]]
    

#### 普通詞的處理原則

普通名詞本身通常不建立 wiki-link。  
但若該詞在本章中承載象徵、隱喻、審判、復興、盟約或救恩意義，應改為連結「完整概念短語」，而不是單一普通詞。

例如：

- 不連 [[弓]]，而連 [[折斷以色列的弓]]
    
- 不連 [[馬匹]]，而連 [[不靠軍事力量得救]]
    
- 不連 [[海沙]]，而連 [[亞伯拉罕之約的海沙應許]]
    
- 不連 [[兒女]]，而連 [[象徵性命名]]
    
- 不連 [[淫亂]]，而連 [[屬靈淫亂]]
    

#### 關於「神、耶和華、主」的處理

不要機械式地把每一個「神」、「耶和華」、「主」都連成同一個 wiki-link。  
但若該詞與特定神學概念、盟約行動、啟示、屬性或救恩作為有關，應建立主題型連結。

例如：

- [[耶和華的話臨到]]
    
- [[耶和華是以色列的丈夫]]
    
- [[神的公義與慈愛]]
    
- [[神的憐憫]]
    
- [[神的棄絕非永遠]]
    
- [[不作你們的神]]
    
- [[我是自有永有的]]
    

#### 連結密度原則

1. 同一段落中，同一詞第一次出現時建立 link 即可，不必每次重複連。
    
2. 標題、表格欄位、主題清單、整合觀察、待確認事項中的關鍵詞應優先建立 link。
    
3. 經文檔可以偏向保守，只連明確人物、地點、國家、書卷與核心主題。
    
4. 註解、解說、拾穗檔應更積極建立主題型連結。
    
5. 若某個概念值得保留但資料不足，可以先建立候選連結，並在待確認事項標註「需補充」。

## 九、強制連結驗證流程（關鍵規則）
**為確保知識網絡 100% 閉合，所有 wiki-link 必須有對應的本地 markdown 檔案。嚴禁依賴 agent「記憶」或「意識」來補齊連結，必須執行確定性腳本驗證。**

### 執行規則（每完成一章必須執行）：
1. **掃描**：跑 `verify_links.py` 掃描該章所有四類檔案（經文、註解、拾穗、解說）中的 `[[...]]`
2. **比對**：與 `人物/`、`地點/`、`主題/` 現有檔案清單比對
3. **列出缺失**：輸出 `missing_entities.txt`（格式：實體名|建議類別|引用章節）
4. **補建**：對每個缺失實體，用 `write_file` 直接在對應資料夾建立 markdown 檔案
5. **補建檔案也要補 link**：新建的實體檔案本身也是 wiki 檔案，裡面的 `[[...]]` 也必須在撰寫時就加好
6. **補建後驗證新檔案**：每新建一個實體檔案，立刻對該檔案跑一次 `verify_links.py` 全掃描，確認該檔案內部的連結也都存在
7. **再驗證**：再次跑 `verify_links.py` 確認 0 破損
8. **才能 commit**：只有驗證通過才可 git commit + push

### ⚠️ 補建實體檔案的品質要求（2026-06-24 教訓）

**禁止：** 用 web_search 搜到的薄泛資料湊數，寫出 2-3 行就交差的 stub。

**要求：**
1. **從指定來源抓取**：entity 資料也應盡量從 kingcomments.com、biblehub.com、ccbiblestudy.org 等指定來源抓取，或從至少兩個可靠來源交叉驗證
2. **完整模板**：必須包含 scheme.md 中定義的所有欄位（基本資料、聖經出現、神學意義、參見、應用與默想、參考資料）
3. **最小內容**：
   - 人物：名字字義、出身背景、主要事蹟、與本書卷的關聯、神學意義
   - 地點：名稱字義、地理位置、歷史背景、與本書卷的關聯、神學意義
   - 主題：定義、原文意義、主要出處經文、神學意義、相關主題
4. **連結完整**：檔案內提到的所有實體都必須有 `[[...]]` link
5. **有深度**：不是百科式的列舉，而是要有**與本書卷的關聯性分析**（例如「亞哈斯在何西阿書中的角色」而非「亞哈斯是猶大王」）
6. **驗證閉環**：補建完一個檔案後，必須對該檔案跑 verify_links.py 確認 0 破損
7. **已有實體不跳過**：遇到已存在的實體，仍要根據本章內容補充其「聖經出現」和「神學意義」，逐章累積深度
